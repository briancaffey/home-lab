# 编队的其他成员

家里并不是每台电脑都是集群节点。集群之外的机器同样影响着它，其中一台可以说是整个系统里最重要的机器。

{/* screenshot: hardware/fleet-family-photo.jpg — Brian-provided photo slot */}

## MacBook Pro —— 运维用的机器

我的日常主力机，并且**刻意不加入集群**。这是操作整个实验室的机器：它保存着 Git 检出、kubeconfig、整个 `.lan` TLS 体系所信任的证书颁发机构（CA），以及让 AI 智能体能够打开密码保险库的 Keychain 凭证。Claude Code 对集群做的所有工作都是从这里进行的。

让它独立于集群之外是有意为之的。集群应该在自己任何一台机器故障后继续运行，而运维用的机器应该在集群故障后继续可用。这样在修东西时总有一个可用的立足点——[破窗应急手册](/gitops/the-trio)的前提就是这台机器存在。它确实让这台笔记本成为自身的单点故障，所以它独有的每一样东西都在被逐项复制进保险库。

它可以加入集群，但不会：一台用来控制和修复集群的机器，不应该同时受制于集群自身的重启和维护。

## Razer Blade 16 —— 备用 GPU 笔记本

一台 GPU 笔记本，目前对实验室没有任何贡献。如果我需要一个突发算力节点——第六台机器，干一票重活就退出——它就是显而易见的候选；或者当一台不进集群的 Windows 游戏机。眼下它是未分配的备用产能。

## 作为节点的 X1 Carbon

我三台笔记本里有一台是集群节点：[x1](/hardware/nodes)。改造它需要做几处改动：免密 sudo、禁用休眠和合盖挂起、关闭 WiFi 省电、安装信任证书，以及一个电池告警。笔记本可以作为服务器运行，但它在一些方面仍然表现得像笔记本，这些是你必须考虑到的。

## 树莓派 —— 未使用

一小摞树莓派，目前没有在用。它们可以承担的最有用的工作：

- **第二台 DNS 解析器** —— 全家现在依赖单节点上的单个 Pi-hole。一台处于独立故障域的树莓派是标准解法，也是把路由器 DHCP 指向 Pi-hole、实现全屋接管的前提。
- **看门狗** —— 一台永远在线的小机器，负责 ping 主要机器并发送网络唤醒（WoL）报文，可以在 spark 离线时唤醒它。
- **异地备份目标** —— 放在亲戚家的一台树莓派，每晚接收 restic 快照，是避免把所有备份放在同一栋楼里的最便宜方案。

这些目前都还不存在，但每一个都是一个小项目。

## 愿望清单

1. **网线布线** —— 最高优先级。每个节点都在 WiFi 上，这个实验室里每一个难题（存储复制、CI 传输速度、大镜像拉取、spark 的不稳定）都会因为网线而变得更容易。这是手头最大的单项改进，成本很低。
2. **第二台 Pi-hole / DNS 故障域** —— 见上；同时启用全屋去广告。
3. **异地备份** —— 灾难恢复计划中剩下的缺口。
4. **给 a3 加硬盘** —— 控制平面的数据盘是全编队里最满的。
5. **第六个节点，未来某天** —— 但现有编队还没用满，所以下一个升级是网线，而不是更多电脑。
