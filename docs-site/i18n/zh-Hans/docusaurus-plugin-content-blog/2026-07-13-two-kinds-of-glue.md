先说实在话：几天前，我给了实验室一个*按计划跑代码*的地方——[Dagster](/data/dagster)。这周，我又给了它另一半——一个*对事情做出响应*的地方。那就是 [n8n](/data/n8n)，一个自托管的工作流自动化工具（可以想成 Zapier 或 Make，但是我自己的），它就住在"数据 / 编排"这个新类别里、Dagster 的旁边。它俩把"跑一段逻辑然后做点事"这件事干净利落地一分为二：Dagster 是定时数据流水线，n8n 是事件驱动的 webhook 与集成黏合层。

## 为什么两个编排器不算多了一个

这事我来回想了好几遍。Dagster 不就是*那个*编排层吗？为什么还要再加一个？答案是：它俩是为不同的问题塑形的。Dagster 极擅长"每天早上把这些资产材料化"——一张你希望存在的东西的依赖图，按计划运行，每次运行都是它自己的 Kubernetes Job。但它是"*当这个 webhook 触发时，就去做那件事*"的错误工具。响应式、重集成、一次性的胶水活儿，想要的是另一种形状：一块摆满预置服务节点的画布、一个触发器、外加一个拖拽拖不动时用的 Code 节点。那就是 n8n。

所以心智模型是：**Dagster = 时钟，n8n = 门铃。** Dagster 醒来是因为到点了。n8n 醒来是因为发生了什么。它俩合起来覆盖了自动化的两半，而我宁可要两个各自称职的工具，也不要一个被硬拉着横跨两边的工具。

## 管道部分很无聊，这是故意的

这是我暗暗自得的一部分。n8n 的部署几乎和 Dagster 一模一样：一个社区 Helm chart 通过 kustomize 展开、一个 Argo CD 应用（`home-n8n`），以及它自己那个朴素的 `postgres:16-alpine`，而不是 chart 自带的子 chart——一样的自带 Postgres 惯例、一样固定到 a2，因为 local-path PVC 就在那儿。我没有发明一套新的部署故事；我复用了上周那套。这正是先把机器造好的全部回报：第二个编排器只花了第一个的零头。

## 有意思的部分：它接到了 LiteLLM 上

n8n 挣得起它这碗饭的地方，是连接组织。main Pod 带着一个通往集群内 [LiteLLM 网关](/ai/litellm)的把手，我还在 UI 里建了一个对应的 OpenAI 兼容凭据。于是任何工作流都能调 LiteLLM 挡在前面的*任何*模型——我自己 GPU 上的本地 vLLM，或者任务大到家里装不下时的某个云模型——都经由其他每个消费者都用的那同一扇门。n8n 不知道模型；它知道一个网关，而网关知道一切。

把这个叠在两百来个内建集成节点之上——Paperless、Immich、Forgejo、MinIO、Mailpit、那个已经给我手机报警的 Telegram 机器人——n8n 就成了一层能对事件做出响应、用 LLM 思考、再作用于家里任何服务的东西。这就是黏合层的定义，也是为什么它和 Dagster 待在同一个类别里，尽管它干的是相反的那种活儿。

## 一课我在不得不学之前就先记下了

n8n 用一个加密密钥加密每一份存下来的凭据，而这个密钥一旦变了，每一份保存的凭据就变成无法解密的垃圾——再怎么恢复 Postgres 都救不回来。chart 会在安装时乐呵呵地生成一个全新的密钥，这是个在你下次重新部署时安静地丢掉一切的法子。所以密钥住在一个 GitOps 环被告知永不重新生成的带外 secret 里，Vaultwarden 里还留了一份副本。Postgres 密码和 LiteLLM 密钥享受同样的待遇。这是实验室最老的规矩换了身新衣裳：secret 从不住在 git 里，而自动化被告知别去跟创建它的人对着干。

## 一个要老实说的缺口

n8n 的原生 git 源码管理——那个能让我把工作流留在仓库里的功能——是企业版专属。社区路径是 Public API，我已经把它打开了，计划是像 [dagster-pipelines](/data/dagster-projects) 已经在做的那样，通过 CI 把工作流 JSON 推进推出一个 Forgejo 仓库。那套环还没建；现在我的工作流只住在 Postgres 里，每夜备份。可以在上面搭建了，但还不到能叫它 GitOps 的时候。把它停在这儿是个诚实的位置——门开着，路也画好了，而实验室里那些好东西，通常就是这么起头的。
