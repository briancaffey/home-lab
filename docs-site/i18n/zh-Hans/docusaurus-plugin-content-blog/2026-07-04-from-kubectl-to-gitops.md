先说实在话：三天前，往我的家庭集群部署任何东西，都意味着回忆那个服务专属的咒语、敲进终端、然后祈祷自己没记错。今天，我合并一个 Pull Request，集群就会自己更新。如果你也玩家庭实验室，而且一直在拖延"正儿八经地上 GitOps"这件事，这篇文章就是我在告诉你：水温刚好，下来吧——而且真正的回报不是自动化，是*可见性*。

## 从前的日子

我的集群是四台（有时五台）跑 k3s 的机器：三台插着 RTX 4090 的主机负责 AI 推理，一台笔记本，还有一台 DGX Spark。上面跑着大约四十个服务：媒体服务器、照片备份、密码保险库、Git 锻造场、模型服务、各种各样的仪表盘。

每个服务在我的仓库里都是一个 Kubernetes 清单目录，每个都靠手动部署：`kubectl apply -k clusters/home/<service>`。除了那些 Helm 的——它们需要带着一字不差的参数执行 `helm upgrade`——而如果你不小心对它们用了 `kubectl`，对象就会落进错误的命名空间，和真正的 release 打起来。（我知道这一点，是因为它真的发生过，还让我的仪表盘挂了一晚上。）

更深层的问题不在于打字，而在于*没有任何东西在盯着*。仓库说一套，集群做另一套，两者之间的鸿沟在东西坏掉之前完全不可见。

## "为什么会有六个 Application？"

迁移是从小处开始的：一个我当试点搭起来的 Argo CD 实例，盯着一个玩具仓库。然后 Claude——和我一起运营这个实验室的 AI 代理，其他文章里会细说——开始迁移真正的服务，一次一个目录。

我第一个真正犯迷糊的时刻来得很早。我以为会是"一个仓库、一次 apply"，结果盯着六个各自独立的 Argo *Application* 问这是怎么回事。答案后来被证明就是整个设计哲学：每个 Application 都是一个拥有独立爆炸半径的独立哨兵。一次糟糕的提交只会搞坏*一个*服务，而不是整个集群。同步策略变成了信任等级——新服务不许删除任何东西（`prune: false`），久经考验的可以，而那些吓人的（下文细说）永远不会在没有人类参与时自我修复。

想通了这一点，剩下的就是节奏问题。

## 一波接一波

迁移以编号任务的形式在一条 Forgejo issue 上推进，那条 issue 后来成了某种船长日志。大致的形状是：

```mermaid
graph LR
    A[推送镜像<br/>Forgejo → GitHub] --> B[Renovate<br/>每晚 PR]
    B --> C[App-of-apps 根应用]
    C --> D[迁移波次<br/>简单 → 媒体 → 推理]
    D --> E[循环依赖三人组]
    E --> F[Longhorn 大典]
    F --> G[迁移看板清空]
```

- **管道铺设**：一个推送镜像让自托管的 Forgejo 和 GitHub 自动保持同步，再加上 Renovate——一个在任何钉定的镜像版本有更新时开 PR 的机器人。我的依赖更新从考古工作变成了代码评审。
- **App-of-apps 根应用**：一个盯着"Application 定义目录"的 Application。从此以后，把一个服务纳入 GitOps 就等于*提交一个文件*。我自己独立完成过一次，当作训练圈：推一个提交，看着新的 Application 凭空出现，全程没人执行 apply。由衷地愉悦。
- **波次推进**：先是简单的无状态服务，然后是媒体服务（漂移检查做得格外仔细——自我修复会回滚你早就忘了自己动过的现场改动），再然后是推理舰队，附带一条关键规则：GitOps *无视*副本数，因为我整天停放和唤醒模型服务器，git 没资格替我唤醒它们。
- **吓人的留到最后**：GitOps 环路自身依赖的那几个服务，还有存储层——各有各的仪式，各是另一篇文章的故事。

## 机器发现了什么

接下来是我没料到的部分。迁移后不到一天，Argo 的健康检查就翻出了两件其他任何东西都没抓到的事：我的媒体服务器已经安静地崩溃循环了**三天**，一个开发部署已经死了**十三天**。它们一直在无声地失败，而从外面看，它们那个小角落一切正常。

这才是真正的卖点。以前，"所有东西都在正常跑吗？"这个问题没有答案，除非手动检查四十样东西。现在，一条命令——`kubectl get applications -n argocd`——任何红色的东西想骗我，都得当着我的面骗。

## 诚实的账本

我得把这一切实际是怎么发生的说清楚，因为这本身就是故事的一部分：Claude 主驾。这个代理写 Application 清单、跑漂移检查、发现并修复事故、把每一步记录在追踪 issue 上。我负责做决定——迁哪些服务、定什么风险等级、什么时候合并——以及点那些合并按钮。三天这样的节奏，产出了我一个人三周也绝对做不出来的东西，而且——更重要的是——每个决定都写在 issue 日志里，未来的我可以重建出每一个*为什么*。

迁移看板现在空了。集群里的每一个工作负载都由 git 驱动。今晚的 Renovate 例行运行提出某个镜像升级时，我会读一份 diff、点一下合并，然后正确的那个 Pod 会自己滚动更新，而我在做别的事。

这就是全部的推销词：不是机器人替你部署软件，而是你终于知道你的集群在干什么——因为要看的地方只有一个。
