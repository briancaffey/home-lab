home-cluster · security audit1 — Overview

🔴 Critical

CriticalC1

Headlamp runs as no-login cluster-admin, exposed over Tailscale

What
Headlamp authenticates every visitor as a pod ServiceAccount bound to cluster-admin, with no login prompt. The values file is commented "LAN-only" — but the Tailscale Ingress right beside it publishes it at headlamp.<tailnet>.ts.net. Confirmed live: binding headlamp-admin → cluster-admin.
Why
Anyone who reaches that tailnet host — any tailnet member, any device that ever receives a tagged key, or one ACL slip — gets full read/write control of the entire cluster, no password. Highest-impact finding: game-over if reached.
Where
clusters/home/headlamp/values.yaml clusters/home/tailscale/ingress-headlamp.yaml
Fix
Remove ingress-headlamp.yaml from the tailscale kustomization (keep it .lan-only), and set clusterRoleName: view + disable unsafeUseServiceAccountToken. Recommended: do both. ~5 min

🟠 High

HighH1

Private registry: no auth, plaintext, push & pull open on the LAN

Why
NodePort 30500, no htpasswd, insecure_skip_verify, reachable on every node IP. Anyone who reaches a node can push an image — a poisoned image is direct code execution on the cluster (every node pulls from it).
Where
clusters/home/registry/registry.yaml
Fix
Add an htpasswd Secret (basic auth); stop exposing it past where it's needed — pushes should come from CI. ~30 min
HighH2

Vaultwarden allows open signups and is on the tailnet

Why
This is the app that stores every other credential. SIGNUPS_ALLOWED:"true" + SIGNUPS_VERIFY:"false" means anyone who reaches it can create an account on your vault.
Where
clusters/home/vaultwarden/vaultwarden.yaml:79 clusters/home/tailscale/ingress-vaultwarden.yaml
Fix
Set SIGNUPS_ALLOWED:"false" now that your account exists; flip back on only transiently to add a user. ~2 min
home-cluster · security audit2 — Critical & High

🟡 Medium — architecture & hygiene

MedM1

No NetworkPolicies

0 policies across 28 namespaces — any pod can reach any service. One compromised :latest image can hit Vaultwarden's DB, the registry, MinIO, Postgres.

Fix: default-deny + allow on the 2–3 crown jewels (vaultwarden, registry, platform Postgres). No mesh needed.
MedM2

Secrets unencrypted at rest

a3 is the sole control plane on SQLite, secrets base64-only. Whoever gets a3's disk or a raw backup gets every secret — and it's a single API SPOF.

Fix: k3s secrets-encrypt enable; pair with the planned HA/etcd migration.
MedM3

No managed secrets

No Sealed/External Secrets/SOPS. Secrets live only in live SQLite + your head — the GitOps repo isn't actually reproducible.

Fix: Sealed Secrets — encrypted secrets become safe to commit. Great "next infra" project.
MedM4

:latest images

16 of 70 images float on :latest (invokeai is untagged). Non-reproducible; a re-pull can change or poison the image; no rollback.

Fix: pin a tag/digest per image; internet-facing first.
MedM5

Tailnet IPs in public repo

docs/07:19-22 lists real 100.x tailnet IPs (→ VNC) — the repo's own stated red line. Not exploitable (CGNAT) but should be scrubbed.

Fix: replace with node names / <tailnet>.
MedM6

No Pod Security baseline

No namespace carries a pod-security.kubernetes.io label — nothing stops a future manifest running privileged/hostPath by accident.

Fix: enforce: baseline on app namespaces; leave GPU/monitoring at privileged.
MedM7

NodePort sprawl (14)

14 services on every node IP — incl. unauthenticated prometheus, minio console, and litellm (master key). Gated only by "the LAN is trusted."

Fix: prefer .lan/Tailscale Ingress; drop the NodePort once an Ingress exists (Phoenix is the template).
MedM8

Root containers

~10 of ~44 workloads set any securityContext; a few are explicitly runAsUser:0. A container escape starts as root.

Fix: add runAsNonRoot + seccompProfile: RuntimeDefault on new services. Low priority on an isolated LAN.
