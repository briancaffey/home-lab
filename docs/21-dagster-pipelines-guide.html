Dagster, in plain terms
the programmable glue that runs code I write, on a schedule, against my own platform

Dagster is the lab's data orchestrator — the first service whose whole job is to use the other services. You write pipelines as Python assets (a thing you want to exist — a table, a report, a file), declare their dependencies, and Dagster works out the order, runs them on a schedule or trigger, records every run in Postgres, and gives you a UI to watch it all. It's in the Airflow family but asset-first and far nicer to look at. Mine lives on a2 at dagster.lan, and the neat part is every run executes as its own Kubernetes Job.

Dagster in one minute (the whole mental model)

You only need three nouns:

  • Asset — a materializable output, an @asset function. Its arguments are its dependencies, so the DAG is inferred from the function signatures — no wiring by hand.
  • Job — a selection of assets you run together (define_asset_job).
  • Schedule / Sensor — when a job runs: a cron, or a reaction to an event.

The one idea that makes everything click

Split the world in two: platform = the home-lab repo (clusters/home/dagster/ — UI, daemon, Postgres, run launcher, deployed by Argo) and pipeline code = the dagster-pipelines repo (my assets, built into a container image and loaded over gRPC). You edit the platform rarely and the pipeline code often. Everything below follows from that split.

"An orchestrator needs a big always-on worker pool." — Unlearn this

Nothing runs inside a long-lived worker here. The K8sRunLauncher asks the cluster for a fresh pod per run and hands execution straight to the scheduler — you watch them come and go in kubectl get jobs -n dagster. It's a genuinely nice demonstration of k8s-native orchestration: the orchestrator delegates, the cluster executes.

Two sharp edges worth writing down

Postgres password ownership. The chart wants to generate its own password Secret, which would let Argo clobber the out-of-band dagster-postgres-secret on every sync. Set generatePostgresqlPasswordSecret: false — the credential is created once, out of band, and the chart is told to keep its hands off (secrets never live in git).
Why everything is pinned to a2. Run and code images live in harbor.lan/apps/, and pulling them requires trusting the Harbor mkcert CA. a2 is Harbor-trusted, amd64, has the roomy e-disk, and co-locates with Postgres — the natural home. (t430 can't host run Jobs yet only because it doesn't trust the Harbor CA — a known follow-up.)

Access & auth — mind the exposure

dagster.lan gets a Traefik ingress with the mkcert dagster-tls cert and is Homepage-discovered under the new Data / Orchestration group; it's also exposed remotely at dagster.<tailnet>.ts.net via the Tailscale operator. Critical caveat: Dagster OSS has no built-in authentication — no login screen. Access is gated entirely by being on the LAN or the default-deny tailnet. Acceptable for a single-operator lab; never expose it more broadly without an auth proxy in front.

War story — the digest that named a GPU pod DRIVERS_LICENSE_1

The first digest described utilization for a pod called DRIVERS_LICENSE_1. Nothing was broken — that was Rampart, the PII-redaction guard, doing its job. Every prompt to LiteLLM passes through Rampart, and a hashed pod name looks enough like an ID number that the redactor swapped it for a placeholder before the model saw it. Great proof the security layer is really in the path; the obvious next refinement is normalizing pod names before they hit the gateway.

The workflow for a new pipeline

Same GitOps machinery as every other app in the lab — push code, CI builds, Argo deploys — with one Dagster-specific twist: promoting the image tag. The platform lives in the home-lab repo; the pipelines live in brian/dagster-pipelines on Forgejo.

1

Write the module

New dagster_pipelines/<name>.py — the assets, plus a new resource in resources.py if it talks to something new.

2

Wire it into definitions.py

Add a job and, if it's scheduled, a schedule.

3

Validate locally

dagster definitions validate -m dagster_pipelines.definitions — catches import and wiring errors before CI.

4

Push → Forgejo Actions builds the image

Pushing the dagster-pipelines repo triggers the runner (a2, dind) to build and push harbor.lan/apps/dagster-pipelines:<sha>.

5

Promote — bump the tag in values.yaml

Set the dagster-user-deployments image tag in clusters/home/dagster/values.yaml to that <sha> and push home-lab. Argo rolls the code server onto the new image.

6

Verify with a K8s-Job run

The new code location loads in the UI; launch the job and watch its Kubernetes Job appear in kubectl get jobs -n dagster.

The version rule CI enforces for you

Pin dagster==1.13.13, and remember the integration libraries use the 0.X.Y scheme that trails core by a fixed offset — so dagster-postgres / dagster-k8s are 0.29.13 for core 1.13.13. They have to be in the image or the run pods fail to import.

Pin an explicit tag, don't chase :latest

Promoting by pinning the exact <sha> (rather than following a moving :latest) keeps deploys reviewable in git and trivially rollback-able — the same discipline as every other GitOps app in the lab.

Conventions to keep

A

Resources for anything external

Put URLs and clients in resources.py, not in assets. Assets stay pure and testable; the cluster seams live in one place.

B

Degrade gracefully

Never fail a run because an optional dependency (the LLM, a flaky API) is down — catch it and fall back, like cluster_summary does.

C

Rich UI metadata

Emit Markdown tables via context.add_output_metadata so a run tells its story in the UI, not just in logs.

D

Local models by default

LLM steps default to the local nemotron-omni (set via LITELLM_MODEL — no rebuild to switch). Cloud models route through the Rampart PII guard, which can redact identifiers.

E

No auth — mind the exposure

Dagster OSS has no login. Fine on the LAN and default-deny tailnet; never put it anywhere more public without an auth proxy.

Bottom line

Dagster is the lab's programmable glue — the layer that runs code you write, on a schedule, against the platform you already built. The platform is a handful of pods in clusters/home/dagster/ deployed by Argo; the pipelines live in dagster-pipelines and ride the same Forgejo → Harbor → Argo loop as every other app, with one twist (promote the image tag). The GPU digest proves the whole loop end-to-end: Prometheus in, LiteLLM narrative out, shipped through your own forge. And for the next pipeline, you don't run the six steps by hand — you describe the idea and let the dagster-project skill do it.
