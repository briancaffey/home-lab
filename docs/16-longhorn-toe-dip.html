The big question, answered up front

"Can two GPUs serve one downloaded copy, and can I still reach the files from plain Docker or a venv on the host?" — YES.

A Longhorn RWX volume (backed by Longhorn's internal NFS share-manager) lets multiple pods on different nodes mount the same filesystem at the same time — and because it's just NFS behind a ClusterIP, the host can mount it too, at /mnt/models, and point HF_HOME there. Same bytes for k8s pods, vanilla Docker, and venv vLLM. Zero re-downloads.

Honest nuance — read before you fall in love

The files are not physically copied to every node's disk. They live on 2 replicas and are served over the LAN to everyone else. Two different mechanisms working together here: replication = durability + mobility (2 live copies, docs/13 p.1–2); RWX-NFS = simultaneous access (one NFS server in front of the volume). This plan deliberately uses both.

Speed check (so nothing surprises you)

The replicas live on spinning HDDs (p.2) — and that costs nothing here: these WD Blues read sequentially at ~150–180 MB/s, while real-world WiFi moves ~25–50 MB/s — and tonight every node is on WiFi (p.3). So the network is the bottleneck either way: loading 15 GB ≈ 5–10 min vs ~30 s from local NVMe. Fine for this test; docs/13 p.3 explains when not to use RWX for weights. Model load time ≠ inference speed — once weights are in VRAM, tokens/sec is identical.

Guardrails — what this is NOT

  • Not a migration. Grafana, Prometheus, Harbor, Forgejo, Paperless — every existing PVC stays on local-path, untouched.
  • longhorn is NOT made the default StorageClass. Nothing new lands on it by accident.
  • No backups yet. Backup target + schedules are issues #18/#21, after this experiment.

SMR vs CMR — the one thing to check on any spinning drive

CMR (conventional) writes every track independently — sustained random writes and full-replica rebuilds are fine. SMR (shingled) overlaps tracks like roof tiles: sequential writes are OK, but sustained random writes — and exactly the full-replica rebuilds Longhorn does — collapse to a crawl once the drive's CMR-style cache zone fills. This is the infamous WD Red NAS debacle. Decoder for this fleet: WD60EZAX = CMR — luckily both drives the plan uses (the a2 + a3 6 TB) are this one; WD60EZAZ / WD40EZAZ = SMR (a1's big drive, a2's NTFS 4 TB) — fine for read-mostly bulk, last pick for anything write-hot.

Disk placement · prereqs

Where Longhorn's disks live — never on a root partition

Principle: Longhorn never touches a root partition in this plan. Left alone, it puts its disk at /var/lib/longhorn — on the root filesystem. We override that with per-node disk config pointing at the big ext4 CMR HDDs:

# disk config comes from an annotation instead of the /var/lib/longhorn default:
kubectl label node a2 a3 node.longhorn.io/create-default-disk=config
kubectl annotate node a2 node.longhorn.io/default-disks-config=\
  '[{"path":"/home/brian/e/longhorn","allowScheduling":true}]'
kubectl annotate node a3 node.longhorn.io/default-disks-config=\
  '[{"path":"/mnt/d/longhorn","allowScheduling":true}]'

ext4 is all it needs — no reformat, no dedicated partition

A Longhorn "disk" is just a directory: replicas are plain sparse files on the existing ext4 filesystem, coexisting with whatever already lives on the drive (Docker data, downloads). Only the NTFS drives are unusable until reformatted.

Disks on exactly two nodes ⇒ every replica-2 volume lands one copy on a2's HDD, one on a3's. Deterministic — no scheduler surprises, no "which node got the replica?"

a1, revisited — optional bulk member, last in line

For write-once/read-many bulk volumes (model files — exactly what its 4.8 T-free ext4 drive could hold), a1 is plausible later: no real-time write path to wait on, and if a replica there rots, Longhorn rebuilds elsewhere and the workload moves. Honest caveats: its USB WiFi makes it the most likely node to trigger rebuild churn, and its HDD is SMR — rebuilds to and from it are slow. Verdict: after the pattern is proven, not during it.

Network audit note: nobody is wired — a2 eno2, a3 eno2, x1 enp0s31f6 are all NO-CARRIER; every member rides WiFi. Fine here: the one heavy event is the ~15 GB download mirroring to replica 2 — tens of minutes, once. Ethernet stays the biggest replication-performance lever (p.7) — a lever, not a gate.

Install · helm release

Install Longhorn — a Helm release, not kubectl

What we do: label the 3 members, then helm install latest stable 1.x into longhorn-system, with committed values that fence every Longhorn component onto a2/a3/x1. Why Helm: same rule as Homepage in CLAUDE.md — the chart is a managed release; applying rendered YAML with kubectl creates namespace drift and fights future upgrades.

# components fence — a2/a3/x1. (Disk placement was configured on p.3: a2 + a3 only.)
kubectl label node a2 a3 x1 inference-club.com/longhorn=member

helm repo add longhorn https://charts.longhorn.io && helm repo update
helm install longhorn longhorn/longhorn \
  -n longhorn-system --create-namespace \
  -f clusters/home/longhorn/values.yaml     # values are committed to the repo

clusters/home/longhorn/values.yaml — the four decisions

defaultSettings:
  defaultReplicaCount: 2                       # 2 live copies per volume
  createDefaultDiskLabeledNodes: true          # honor p.3's labels — never /var/lib/longhorn
  systemManagedComponentsNodeSelector: "inference-club.com/longhorn:member"
persistence:
  defaultClass: false                          # longhorn SC exists but is NOT default
longhornManager:
  nodeSelector: { inference-club.com/longhorn: member }
longhornUI:
  nodeSelector: { inference-club.com/longhorn: member }

Both selectors matter: longhornManager covers the DaemonSet, systemManagedComponentsNodeSelector covers everything Longhorn spawns itself (instance-managers, share-managers, CSI pods). Result: nothing lands on a1 or spark.

UI at https://longhorn.lan

Standard pattern: Traefik ingress + home-tls wildcard cert + gethomepage.dev/* annotations so it appears on Homepage — same as every other .lan service. Ingress manifest lives next to the values file.

The Longhorn UI ships with NO auth — and it can delete volumes

Anyone who can reach longhorn.lan can detach or delete volumes with two clicks. Acceptable on the LAN for a toe-dip; ☐ follow-up: Traefik basic-auth middleware before anything precious lives on Longhorn. Do not expose over Tailscale.

First volume · the model

One RWX volume, one download, two vLLM servers

What we do: a new service dir services/qwen36-nvfp4/ with three pieces — a PVC, a one-shot download Job, and two vLLM Deployments. Why: the PVC's ReadWriteMany mode is the whole trick — creating it makes Longhorn spin up a share-manager pod (the NFS server) automatically. No NFS to configure by hand.

(a) The PVC

apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: models
spec:
  accessModes: [ReadWriteMany]  # ← RWX
  storageClassName: longhorn
  resources:
    requests: { storage: 40Gi }

(b) The download Job — once, on a2

spec:
  template:
    spec:
      # a2 = the 5.3 T HDD; never a1
      nodeSelector:
        inference-club.com/box: a2
      containers:
      - name: fetch
        image: python:3.12-slim
        env: [{ name: HF_HOME, value: /models }]
        command: ["bash","-lc",
          "pip install -U 'huggingface_hub[cli]' &&
           hf download nvidia/Qwen3.6-27B-NVFP4"]
          # ↑ confirm exact repo id at download time

Host access · the big question

Mount the same volume on the host — step by step

What we do: find the NFS endpoint Longhorn created for the RWX volume, then NFS-mount it on a2's host at /mnt/models. Why it works: the share-manager Service is a ClusterIP, and kube-proxy makes ClusterIPs reachable from every cluster node's host — no NodePort, no ingress, no credentials.

# 1 · find the share-manager Service for the volume (one per RWX PVC):
kubectl -n longhorn-system get svc | grep pvc-
#   pvc-1a2b3c…   ClusterIP   10.43.x.x   2049/TCP

# 2 · on a2's host — mount it:
sudo mkdir -p /mnt/models
sudo mount -t nfs4 -o vers=4.1 <cluster-ip>:/<pvc-name> /mnt/models

# 3 · same files the pods see:
ls /mnt/models/hub/

The two vanilla flows — zero re-download

venv vLLM on the host

HF_HOME=/mnt/models vllm serve \
  nvidia/Qwen3.6-27B-NVFP4 \
  --port 8001

vLLM finds the weights already cached under $HF_HOME/hub and skips the download entirely.

plain Docker on the host

docker run --gpus all -p 8001:8000 \
  -v /mnt/models:/root/.cache/huggingface \
  vllm/vllm-openai:latest \
  --model nvidia/Qwen3.6-27B-NVFP4

/root/.cache/huggingface is the default HF_HOME inside the image — the bind-mount lands the cache exactly where it looks.

Caveats — know these before you rely on the mount

  • umount before deleting the volume. If the PVC goes away under a live mount, the mount hangs hard.
  • The ClusterIP changes if the PVC is ever recreated — re-run step 1, don't hardcode it in fstab.
  • Reads are network-speed — today that means WiFi (p.3), which is slower than the HDDs themselves (p.1). Fine for weight loading.
  • Only works ON cluster nodes (kube-proxy routes ClusterIPs there) — which is where the GPUs are anyway.

Why this matters

This closes the gap between "cluster storage" and "my normal tools." The same 15 GB serves: two k8s vLLM pods, a venv experiment, and a throwaway Docker container — downloaded once, replicated twice, mountable anywhere on the cluster's hosts.

# 2 · Longhorn UI: volume Healthy, 2 replicas — one on a2's HDD, one on a3's.
# 3 · same bytes, pod vs host mount — hashes must match:
F='hub/models--nvidia--*/snapshots/*/model-00001*.safetensors'
kubectl -n qwen36-nvfp4 exec deploy/qwen-a3 -- sh -c "sha256sum /models/$F"
sha256sum /mnt/models/$F                    # on a2's host — hashes must be identical

Optional 5-minute chaos drill — the whole point of Longhorn

In the UI, open the volume and delete one replica. Watch Longhorn auto-rebuild it on a healthy node while both vLLM pods keep serving. That's the durability story from docs/13 p.2, demonstrated instead of believed.

Teardown — in this exact order

# 1 · host mount FIRST (see p.5 caveat):
sudo umount /mnt/models
# 2 · the servers:
kubectl -n qwen36-nvfp4 delete deploy qwen-a2 qwen-a3

Decision point — the PVC. If #7 benchmarking continues, keep it: the weights stay downloaded, replicated, and remountable in seconds. If done, delete it — kubectl -n qwen36-nvfp4 delete pvc models — which reclaims the space on both replicas at once. Longhorn itself stays installed either way; it costs little idle.
